# 代码代理工作规约

## 适用范围

1. 本文件约束本仓库后续所有由代码代理执行的代码修改、文档修改、脚本修改、配置修改和数据适配修改。
2. 每轮实际修改开始前，代理必须先阅读本文件，并据此规划本轮任务。
3. 如果用户的临时指令与本文件冲突，代理必须先向用户说明冲突点并请求确认，不得擅自绕过本文件。
4. 如果修改涉及生产链路、公共接口、数据结构、运行脚本或测试行为，默认按生产代码变更处理。

## 变更原则

1. 生产代码修改默认以安全、可回滚、低耦合、易交接为优先级。
2. 禁止在没有记录的情况下做大范围隐式改动。
3. 禁止把不相关改动混入同一轮任务；重构、行为修改、配置修改、数据结构修改应尽量拆分。
4. 新增逻辑优先通过开关、配置、适配器、包装器、兼容层或专属模块接入，降低回滚成本。
5. 涉及删除旧逻辑时，必须先评估是否可以保留旧路径并逐步切换。
6. 高风险改动必须先记录风险、原因、回滚路径、回退前提和替代方案。
7. 若某项修改天然难以回滚，必须在记录中明确标注，并说明为什么不能采用更低侵入方案。

## 实施前检查

每轮动手前，代理必须识别并记录以下内容：

1. 修改目标：本轮要解决什么问题，不解决什么问题。
2. 直接影响模块：会被直接编辑或调用路径发生变化的文件、模块、脚本。
3. 潜在连带影响模块：可能因接口、导入、配置、数据流、运行入口变化而受影响的模块。
4. 依赖影响：第三方依赖、模型服务、环境变量、数据库、缓存、外部文件是否受影响。
5. 接口影响：函数签名、类构造参数、命令行参数、环境变量、数据库字段、返回结构是否变化。
6. 数据结构影响：输入数据、输出数据、中间结构、持久化格式是否变化。
7. 测试影响：需要新增、修改或运行哪些测试或 smoke 检查。
8. 上线影响：运行方式、配置方式、数据准备方式是否变化。
9. 回滚影响：如何退回旧逻辑，退回时需要恢复哪些文件或配置。
10. 若信息不足以保证安全修改，必须先记录假设和风险，再执行最小必要改动；需要业务判断时应停止并提问。

## 实施方式

1. 优先小步修改，保持每一步都可审阅、可验证、可回滚。
2. 优先保持旧接口兼容；如果必须破坏兼容性，必须记录原因、影响范围和迁移方式。
3. 优先使用 feature flag、环境变量、配置项、适配器、包装器、兼容层或专属模块实现渐进式变更。
4. 除非任务明确要求，不顺手清理无关代码、不做无关格式化、不移动无关文件。
5. 如需重构，必须说明重构与行为变更的边界；能拆开时应拆开。
6. 修改公共模块前，必须先评估是否能在业务专属模块中完成。
7. 修改运行脚本前，必须说明默认行为是否变化；如默认行为变化，应提供旧行为回退方式。
8. 修改配置或环境变量前，必须说明默认值、兼容值和缺失值处理。

## 记录要求

以下情况必须更新记录：

1. 开始一轮实质性修改前。
2. 方案发生变化时。
3. 发现新的风险、边界条件或兼容性问题时。
4. 完成关键验证后。
5. 最终交付时。

记录规则：

1. 默认记录目录为 `worklogs/`。
2. 设计说明、迁移说明、排障说明、运行说明和测试说明可放在 `docs/` 下。
3. 记录必须能追踪到相关文件、相关命令、验证结果和回滚方式。
4. 记录不能只写“已完成”，必须说明改了什么、为什么这么改、影响哪里、怎么验证、怎么回滚。
5. 如果某个关键决策未采用其他方案，必须记录未采用方案和原因。

## 最终 worklog 要求

每轮实际修改完成后，必须产出一个 md 格式最终 worklog。默认命名规范：

1. 默认路径：`worklogs/YYYY-MM-DD-<task-name>.md`
2. `<task-name>` 使用小写英文或拼音短横线命名。
3. 示例：
   - `worklogs/2026-04-12-weibo-agent-profile-audit.md`
   - `worklogs/2026-04-12-weibo-prompt-profile-builder.md`
   - `worklogs/2026-04-12-agent-rules-bootstrap.md`
4. 如果同一天同任务多轮修改，在末尾加序号，例如 `worklogs/2026-04-12-weibo-agent-profile-audit-02.md`。

最终 worklog 至少包含：

1. 背景与目标。
2. 变更范围。
3. 具体改动。
4. 关键决策。
5. 影响评估。
6. 验证方式与结果。
7. 回滚方案。
8. 遗留问题与后续建议。
9. 相关文件列表。

最终 worklog 不能写成流水账，必须具备工程交接价值。

## 可接手性要求

1. 如果后续工程师仅看 diff 难以理解，必须补充说明文件、注释、迁移说明、运行说明、测试说明或排障说明。
2. 判断标准是：不在场的人可以低成本继续推进，而不是只能从头逆向理解。
3. 必要时应主动创建或更新：
   - `docs/` 下的设计说明、迁移说明、排障说明。
   - `worklogs/` 下的阶段记录。
   - 模块内 README。
   - 必要的代码注释。
   - 测试说明。
   - 回滚说明。
   - 运行和验证脚本说明。
4. 注释只用于解释非显然逻辑、兼容边界、回滚开关或风险点，不写无意义复述。

## 结束前自检

每轮结束前，代理必须检查：

1. 本轮改动是否足够独立，是否混入无关改动。
2. 是否具备清晰回滚路径。
3. 是否已经留下必要记录。
4. 是否已经产出最终 worklog。
5. 是否存在影响交接的信息缺口。
6. 是否运行了与风险相匹配的验证。
7. 如果存在缺口，必须先补齐再结束。

最终回复必须明确说明：

1. 新增或修改了哪些文件。
2. `agent.md` 做了哪些补充或约束。
3. worklog 文件路径。
4. 基本回滚方式。
5. 尚未覆盖的风险点。

## 微博项目专项规则

1. 所有微博相关内容、提示词、说明、日志说明默认使用中文。
2. 微博数据以 `weibo_test/total_data_with_descriptions_transformers.json` 为准；如用户提到 Windows 路径，应映射到本仓库对应数据文件。
3. 微博没有的数据不要保留在微博新函数或新 prompt 中，不生造 `MBTI`、年龄等字段。
4. 微博改动优先集中在 `oasis/weibo/` 等微博专属文件中，少改原始公共文件。
5. 原始 OASIS、Twitter、Reddit 链路默认不动；确需修改公共链路时，必须先说明原因、影响范围和回滚方式。
6. 推荐系统暂时沿用现有逻辑，只做最小必要适配。
7. 修改前必须先给计划，待用户明确指令后再实施。
8. 所有代码和数据适配必须围绕当前微博数据集进行，并运行与改动风险相匹配的测试或 smoke 检查。
9. 拿不准的问题不得自作主张；应立即停止并向用户提问。
