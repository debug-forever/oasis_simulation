# 微博智能体建模属性审计报告

## 背景

本报告回答“智能体建模时取了哪些属性”，并说明本轮为什么要补强微博画像输入。数据源固定为 `weibo_test/total_data_with_descriptions_transformers.json`，共 100 条用户记录。

本轮只评估和修改微博新链路，原始 OASIS、Twitter、Reddit 链路不纳入本次变更。

## 当前数据集字段

数据顶层字段包括：

- `用户ID`
- `账号类型与层级`
- `个人基本信息`
- `社交影响力`
- `关注博主信息`
- `近期发帖内容分析`
- `用户行为特征`
- `标签特征`

可用于画像且本轮已接入的字段：

- `个人基本信息.用户名`
- `个人基本信息.用户简介`
- `个人基本信息.用户性别`
- `个人基本信息.微博等级`
- `社交影响力.粉丝数量`
- `社交影响力.粉丝等级`
- `社交影响力.转评赞统计`
- `账号类型与层级.用户认证信息`
- `账号类型与层级.用户层级`
- `关注博主信息.关注数`
- `关注博主信息.follows`
- `近期发帖内容分析.全部帖子合集`
- `用户行为特征.总发帖数`
- `用户行为特征.互动密度`
- `用户行为特征.发帖密度`
- `用户行为特征.活跃天数`

- `用户行为特征.keywords_list`
- `标签特征.内容偏好`
- `标签特征.情感倾向`

## 旧逻辑实际输入给智能体的属性

旧的微博新链路在 `oasis/weibo/generator.py` 内部构造 `UserInfo.profile.other_info.user_profile`，实际主要包含：

- `微博等级`
- `粉丝数量`
- `keywords_list` 中前 3 个关键词名
- `用户性别`

旧逻辑还尝试写入下列内容，但这些内容不适合作为当前微博数据画像入口：

- `年龄`：当前数据集中不存在。
- `地区信息`：当前数据集中 100 条均为 `未知`。
- `raw_record`：整条原始记录被塞进 `other_info`，但默认 prompt 不直接使用，且会增加不可控上下文负担。
- `followers`：当前数据集中不存在，真实关注数据字段是 `关注博主信息.follows`。
- `用户昵称`、`个人基础信息`：当前数据集中不存在，不能继续作为微博新链路入口。

结论：旧逻辑确实读了 `user_profile`，但画像摘要偏薄，且保留了几个无效或不存在的数据入口。

## 本轮补强后的建模输入

本轮新增 `oasis/weibo/profile_builder.py`，由微博专属 builder 统一产出中文画像摘要。补强后的 `user_profile` 会使用下列有效属性：

- 身份与简介：用户名、用户简介、用户性别。
- 账号层级：微博等级、用户认证信息、用户层级。
- 影响力：粉丝数量、粉丝等级、转评赞统计。
- 行为习惯：总发帖数、互动密度、发帖密度、活跃天数、活跃时间分布。
- 兴趣偏好：关键词及频次、内容偏好、情感倾向。
- 社交关系：关注数、关注博主摘要。
- 历史表达：近期发帖样例，最多取 3 条并清理 HTML 标签。

这些内容会进入微博专属中文系统提示词，使大模型能看到更完整的人设、兴趣、行为和社交上下文。

## 明确删除或不进入 prompt 的内容

本轮删除的新链路入口：

- `年龄`
- `MBTI`
- `followers`
- `用户昵称`
- `个人基础信息`

本轮不进入 prompt 的字段：

- `地区信息`：字段存在，但当前全为 `未知`。
- `职业`：字段存在，但当前全为 `未知`。
- `教育背景`：字段存在，但当前全为 `未知`。
- `账号类型`、`认证类型`：当前是数字编码，未确认编码含义；改用人类可读的 `用户层级`、`用户认证信息`。
- `权重系数`：更像算法权重，不适合直接作为用户人设。
- 全量历史帖子：只取前 3 条样例，避免 prompt 过长和上下文污染。
- 整条原始记录：不再放入 `other_info.raw_record`。

## 模型运行优先级

本轮采用以下原则验证模型链路：

1. 优先检查 vLLM：读取 `WEIBO_VLLM_ENDPOINTS` 和 `WEIBO_VLLM_MODEL`，检查本机是否安装 vLLM 或是否有可访问的 OpenAI-compatible vLLM 服务。
2. vLLM 不可用时再检查 API：读取 `.env` 中微博 API 相关配置，但不把密钥写入代码、报告或日志。
3. DeepSeek 普通 `/v1` 接口对工具 schema 中的 strict 字段存在兼容风险，必须以单 agent tool-call smoke 先验证。
4. Qwen/DashScope 走 OpenAI-compatible 接口，也先做单 agent smoke。
5. smoke 通过后再考虑完整批量运行，避免一次性触发大量模型调用。

当前本机检查结论：`.venv` 中有 `camel`、`dotenv`、`openai`，但没有 `vllm` 模块；系统路径也没有 `vllm` 命令。因此若未提供外部 vLLM 服务地址，本轮应走 API 兜底验证。

本轮代码处理：DeepSeek API 示例已在检测到 DeepSeek 地址或模型名时切换到 CAMEL `ModelPlatformType.DEEPSEEK`，用于规避普通 OpenAI-compatible 后端保留 strict 工具 schema 的风险；vLLM 示例已显式使用 `WeiboSocialAgent`，保证 vLLM 优先链路也使用微博专属中文 prompt。

## 影响评估

- 微博 agent 的系统提示词更长，但内容来自真实字段，且最多只取 3 条近期发帖样例，避免全量历史帖进入 prompt。
- `generate_weibo_agent_graph()` 和 `generate_weibo_agents()` 函数签名不变。
- 推荐系统逻辑不变。
- 原始 Twitter/Reddit/OASIS 链路不变。
- 如果需要回滚，只需恢复 `oasis/weibo/generator.py`、`oasis/weibo/agent.py`，并删除 `oasis/weibo/profile_builder.py`。

## 建议

- 交付给甲方时，可以明确说明：当前智能体画像已包含身份、影响力、行为习惯、兴趣偏好、社交关注和近期表达六类信息。
- 如果后续甲方要求更强人设，可优先补“历史发帖聚类摘要”或“关注对象主题摘要”，不要补数据集中不存在的人口统计字段。
- 如果 DeepSeek API 作为主要运行链路，应单独做工具调用兼容层或改用 DeepSeek 专属模型类型，避免 strict schema 兼容问题影响演示。
